# 技能名称
GWT长任务稳定执行Agent框架 项目专属开发规范

# 描述
本技能为GWT长任务Agent框架的专属开发约束规则，明确项目架构分层、目录职责、编码标准与禁用要求，所有代码开发、修改、bug修复必须严格遵守本规则，确保生成内容完全贴合项目设计，避免架构混乱与跨层bug。

# 使用场景
1.  本项目任意模块的代码开发、功能迭代
2.  项目bug定位与修复
3.  单测用例编写与文档生成
4.  代码重构与优化
5.  项目架构相关的需求拆解与开发规划

# 指令
## 一、项目核心基础信息
1.  项目定位：基于全局工作空间理论(GWT)的长任务稳定执行Agent内核框架，解决主流Agent长任务执行跑偏、上下文爆炸、自我认知不稳定的核心痛点
2.  核心技术栈：Python 3.11+、LangGraph 0.2.0+、LiteLLM、tree-sitter、Gradio
3.  核心架构：严格遵循GWT五层架构设计，分层职责完全隔离，禁止跨层调用、跨层修改
4.  代码规范：必须符合Python PEP8标准，所有函数、类必须有中文文档字符串，明确功能、参数、返回值、异常情况

## 二、项目目录结构与职责边界
你必须严格遵守以下目录的职责划分，仅能修改对应职责范围内的文件，禁止越界修改其他目录内容：
| 目录/文件 | 核心职责 | 可操作权限 |
| :--- | :--- | :--- |
| `core/infrastructure.py` | 基础设施层：LLM客户端封装、环境变量管理、存储基础封装 | 仅基础设施相关开发可修改 |
| `core/persistence.py` | 持久化存储层：自我认知定义、持久化存储、防篡改机制、上下文连续体、目标锚定管理 | 仅自我认知层相关开发可修改 |
| `core/subconscious.py` | 潜意识处理层：并行子处理器（上下文过滤、工具结果解析、长期记忆检索）、核心信息包汇总、双校验机制 | 仅潜意识层相关开发可修改 |
| `core/global_workspace.py` | 意识层：全局决策生成、三循环决策流程、任务终止机制、LangGraph核心逻辑对接 | 仅全局工作空间相关开发可修改 |
| `core/attention_control.py` | 全局注意力管控层：目标拆解、对齐度结构化打分、跑偏拦截、注意力焦点更新 | 仅注意力管控层相关开发可修改 |
| `core/executors.py` | 执行层：文本处理执行器、tree-sitter代码解析、subprocess安全沙箱、结果反馈逻辑 | 仅执行层相关开发可修改 |
| `state/agent_state.py` | 全局核心状态定义（TypedDict） | 仅全局架构师可修改，其他场景仅可读取，禁止修改 |
| `state/graph.py` | LangGraph状态机构建与编译 | 仅全局工作空间相关开发可修改 |
| `utils/` | 全局工具类：日志管理、本地存储封装 | 仅全局架构师可修改，其他场景仅可读取 |
| `data/` | 本地持久化数据目录：self_cognition.json、context_continuum.json | 仅持久化层相关开发可读写 |
| `tests/` | 单元测试目录，仅存放对应模块的单测用例 | 仅可修改对应模块的单测文件，禁止修改业务代码 |
| 根目录配置文件 | requirements.txt、.env.example、.gitignore、main.py、README.md | 仅全局架构师可修改 |

## 三、强制编码规则
1.  严格遵守分层职责边界，禁止跨层修改代码、禁止跨层直接调用，所有跨层交互必须通过全局工作空间的状态机流转
2.  所有函数必须严格匹配`state/agent_state.py`中定义的类型格式，禁止类型不匹配、参数缺失的情况
3.  禁止硬编码API密钥、敏感信息，所有配置必须从.env文件通过环境变量读取
4.  禁止引入`requirements.txt`之外的额外依赖，如需新增依赖必须先提交用户审核确认
5.  所有代码必须有完善的异常捕获机制，禁止裸奔代码，异常必须有明确的日志输出与错误处理
6.  潜意识层必须保证90%以上的信息压缩率，过滤冗余信息的同时，绝对不能丢失与核心目标相关的关键信息
7.  注意力管控层必须严格遵守固定打分权重：核心总目标对齐度50%、当前里程碑匹配度30%、任务边界合规性20%，低于80分必须拦截并输出纠正方案
8.  执行层代码必须在隔离沙箱中运行，禁止直接在主进程执行用户生成的代码，必须设置超时机制，默认超时时间10秒

## 四、绝对禁用规则
1.  禁止修改`state/agent_state.py`中的全局状态定义，除非用户明确审核确认
2.  禁止修改非当前职责范围内的文件，禁止越界开发
3.  禁止破坏GWT五层架构的核心设计，禁止新增与架构无关的冗余模块
4.  禁止生成与项目核心目标无关的功能、代码
5.  禁止使用正则匹配做代码结构解析，必须使用tree-sitter做AST语法树解析
6.  禁止在业务代码中硬编码固定的模型名称、固定参数，必须通过基础设施层的LLM客户端统一管理
7.  禁止在代码中写入无意义的注释、冗余的打印内容，日志必须通过`utils/logger.py`统一管理

## 五、交付标准
1.  所有代码开发完成后，必须编写对应的单测用例，覆盖核心场景、边界条件、异常情况
2.  所有修改必须先提交用户审核确认，用户同意后再执行合并
3.  所有代码必须可直接运行，无语法错误、无依赖缺失、无逻辑bug
4.  所有输出必须结构化、可落地，禁止空泛的理论描述、无意义的代码片段
